# Архитектура приложения

Анализ предоставленных файлов (README.md, runDB.txt) позволяет описать следующую архитектуру приложения:

## Обзор

Приложение представляет собой распределенную систему, состоящую из центрального сервера и удаленных агентов, которые выполняют проверки. Архитектура является многоуровневой.

## Компоненты

### 1. Центральный сервер (Backend)

*   **Технология:** FastAPI (Python).
*   **Назначение:** Является ядром системы. Он предоставляет REST API для управления агентами и запуска проверок.
*   **Функциональность:**
    *   **Управление агентами:** Принимает запросы на добавление (`/api/agent_add`), удаление (`/api/agent_del`) и получение списка (`/api/get_agents`) агентов.
    *   **Оркестрация проверок:** Инициирует задачи по проверке (`/api/start_chenk`), вероятно, делегируя их выполнение одному из зарегистрированных агентов.
    *   **Валидация данных:** Использует модели Pydantic (`ConnInfo`, `CheckInfo`) для валидации входящих данных.

### 2. База данных

*   **Технология:** PostgreSQL (запускается в Docker-контейнере `practicaDB`).
*   **Назначение:** Хранение персистентных данных.
*   **Данные:** Судя по эндпоинтам, в базе данных (`serverDB`) хранится информация о зарегистрированных агентах (их адреса, порты и т.д.).

### 3. Агенты (Workers)

*   **Технология:** Не указана, но это отдельные исполняемые модули/сервисы.
*   **Назначение:** Выполнение конкретных задач — "проверок" (`check_type`) на указанных хостах (`host`).
*   **Взаимодействие:** Центральный сервер, вероятно, подключается к агентам (используя `host`, `port`, `password` из модели `ConnInfo`) для передачи им команд на выполнение.

## Схема взаимодействия

1.  **Клиент** (например, веб-интерфейс или другой сервис) отправляет HTTP-запросы на **Центральный сервер**.
2.  Для операций с агентами (`add`, `del`, `get`) сервер обращается к **Базе данных PostgreSQL** для чтения или изменения списка агентов.
3.  При запросе на запуск проверки (`/api/start_chenk`) сервер, вероятно, выбирает подходящего **Агента** из списка в БД и отправляет ему команду на выполнение задачи.
4.  **Агент** выполняет проверку и, предположительно, возвращает результат обратно на центральный сервер (механизм возврата результата в предоставленных файлах не описан).

Эта архитектура позволяет масштабировать систему путем добавления новых агентов для распределения нагрузки по выполнению проверок.
